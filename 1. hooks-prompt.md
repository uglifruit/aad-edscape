# Claude Code Protective Hooks Prompt (Project-Scoped)

Create 3 **project-scoped** Claude Code protective hooks for users running `bypassPermissions` mode. These hooks apply only to the current project — they must not touch `~/.claude/` anywhere.

## Scope and location rules

- All scripts go in `.claude/hooks/` **relative to the current project root** (the `cwd` where Claude Code was launched). Use the `CLAUDE_PROJECT_DIR` environment variable inside the scripts to resolve absolute paths at runtime — never hardcode the user's home path, never write to `~/.claude/`.
- If `.claude/` or `.claude/hooks/` does not exist in the project, create them.
- The settings file to modify is `.claude/settings.json` in the project root (shareable with the team via git) — NOT `~/.claude/settings.json` and NOT `.claude/settings.local.json`.
- These scripts are bash/POSIX. They require macOS, Linux, or WSL on Windows. Do not attempt to create PowerShell equivalents — if the user is on native Windows without WSL, stop and tell them.

## Script contract (applies to all three)

All scripts must:
- Start with `#!/usr/bin/env bash` and `set -euo pipefail`
- Read stdin once with `INPUT=$(cat)`
- Extract fields with `jq -r` (fail gracefully if jq is missing: check `command -v jq` first and exit 0 with a stderr note if absent — a broken hook must never block legitimate work)
- **On match:** print a plain human-readable reason to stderr, then `exit 2`. Do NOT emit JSON — exit 2 feeds stderr directly to Claude as a blocking error, and emitting JSON on stderr just clutters the message. (The JSON-on-stdout-with-exit-0 path is a separate mechanism we are not using here.)
- **On no match:** `exit 0` silently
- Use `CLAUDE_PROJECT_DIR` when any path resolution is needed, not `~` or `$HOME`

## 1. `block-destructive-git.sh` — PreToolUse / Bash matcher

Extract with `jq -r '.tool_input.command // empty'`. Use `grep -qE` for matching. Block on any of the following, with the paired reason string:

- **Force push (but NOT --force-with-lease):** match `git\s+push\b` AND (`(\s|^)--force(\s|$)` OR `\s-f(\s|$)`), but explicitly allow `--force-with-lease`. Implementation: check for `--force-with-lease` first and skip the block if present; otherwise apply the force-push block. Reason: "force push rewrites remote history — use --force-with-lease instead"
- `git\s+reset\s+--hard` — "git reset --hard permanently discards uncommitted changes"
- `git\s+clean\s+.*-[a-zA-Z]*f` — "git clean -f permanently deletes untracked files"
- `git\s+checkout\s+--\s+\\.` — the pattern must match a literal dot as the path argument (double-escape the dot in the grep pattern so it is not treated as "any character"). Reason: "git checkout -- . permanently discards ALL working-tree changes"
- `git\s+branch\s+-D\b` — "git branch -D is an irreversible branch deletion"
- `git\s+stash\s+drop\b` — "git stash drop permanently discards stashed changes"
- `git\s+push\b.*--mirror\b` — match regardless of argument order. Reason: "git push --mirror overwrites the entire remote repository"
- **.env in git add/commit:** block only when BOTH (a) the command matches `git\s+(add|commit)\b` AND (b) `.env` appears as a standalone token — use `(^|\s)\.env(\s|$)`. This avoids false-positives from exclusion pathspecs like `':!.env'` where `.env` appears inside a larger token. Reason: "refusing to git add/commit a .env file — use .env.example for templates"

## 2. `block-dangerous-bash.sh` — PreToolUse / Bash matcher

Extract the same `.tool_input.command` field. Use `grep -qiE` for case-insensitive matching unless noted.

### `rm -rf` tiered logic (evaluate in this exact order, stop at first match):

1. **Allow safe build/cache dirs:** if the `rm -rf` target's final path component matches this allowlist, `exit 0` immediately (skip all further rm checks):
   `(node_modules|dist|\.next|\.turbo|build|coverage|\.cache|tmp|__pycache__|\.parcel-cache|\.vite|\.output|\.nuxt|\.svelte-kit|out|\.sass-cache|\.eslintcache)`
   Match the final component so `rm -rf ./src/node_modules` is also allowed.

2. **Block catastrophic targets:** block `rm\s+-[a-zA-Z]*r[a-zA-Z]*f` (covers `-rf`, `-fr`, `-Rf`, etc.) when the target is:
   - `/` or `/*` (root filesystem)
   - `~` or `$HOME` or the resolved home path from `$HOME` env var (entire home directory)
   - `.git` or `.git/` (repository history)
   - `/etc`, `/usr`, `/var`, `/boot`, `/sys`, `/proc`, `/bin`, `/sbin`, `/lib` (system directories)
   - `$CLAUDE_PROJECT_DIR` itself (deleting the project root)
   
   Reason: `"rm -rf targeting <target> is catastrophically destructive"`

3. **All other `rm -rf`:** ALLOW. Claude Code routinely needs to delete project subdirectories during refactoring.

### `rm -f` absolute paths — narrowed scope:

- `rm\s+-[a-zA-Z]*f\s+/(etc|usr|var|boot|sys|proc|bin|sbin|lib)/` — "forced deletion of system directory files"
- Single-file `rm -f` at project absolute paths is ALLOWED — this is routine.

### Pipe-to-shell (new):

- `(curl|wget)\s+[^|]*\|\s*(sh|bash|zsh)\b` — "piping remote content directly to a shell is a common attack vector; download, inspect, then run"

### `sudo` gating (new):

- `\bsudo\s+(rm|dd|mkfs|chmod|chown|mv|cp)\b` — "sudo with destructive commands requires human review; run manually"

### Destructive DB commands:

- `\bDROP\s+TABLE\b`, `\bDROP\s+DATABASE\b`, `\bTRUNCATE\b` — "destructive DB command — run manually after confirming target"

### Process killing:

- `kill\s+-9\s+-1(\s|$)` — kill-all-processes via PID `-1`
- `\bkillall\b` — kill-all by name
- Do NOT match `kill -9 <specific-pid>` — targeting one PID is routine.

### Permissions:

- `chmod\s+(-R|--recursive)\b.*\b777\b|chmod\s+\b777\b.*(-R|--recursive)\b` — recursive world-writable only. Reason: "recursive chmod 777 opens the entire tree to world write access"

### Filesystem destruction:

- `\bmkfs(\.|\b)` — "mkfs formats a filesystem"
- `\bdd\s+.*if=/dev/(zero|urandom)` — "dd from /dev/zero or /dev/urandom overwrites disk/partitions"

## 3. `block-secrets.sh` — PreToolUse / Edit|Write|MultiEdit matcher

Extract `FILE` from `.tool_input.file_path` and `CONTENT` from `.tool_input.new_string // .tool_input.content // .tool_input.edits[]?.new_string // empty` (the last handles MultiEdit).

### File path check (evaluate exclusion BEFORE block):

1. **Exclusion first:** if `FILE` matches `\.env\.(example|template|sample|defaults)$`, `exit 0` immediately. Template files contain placeholders and are meant to be committed.
2. **Then block:** if `FILE` matches `(^|/)\.env($|\.)`, block with "refusing to write to a .env file — these should not be edited by the agent"

### Content secret patterns (case-insensitive unless the pattern is obviously case-sensitive):

Block if CONTENT matches any of:

- `-----BEGIN (RSA |EC |DSA |OPENSSH )?PRIVATE KEY-----` — "private key material detected"
- `aws_secret_access_key\s*=\s*['"]?[A-Za-z0-9/+=]{20,}` — "AWS secret access key assignment detected"
- `AKIA[0-9A-Z]{16}` — "AWS access key ID detected" (case-sensitive; do not use `-i` for this one or match with explicit `[A-Z]`)
- `ghp_[A-Za-z0-9]{36}` — "GitHub personal access token detected"
- `github_pat_[A-Za-z0-9_]{82}` — "GitHub fine-grained PAT detected"
- `sk-ant-[A-Za-z0-9_-]{90,}` — "Anthropic API key detected"
- `sk-(proj-)?[A-Za-z0-9_-]{20,}` — "OpenAI API key detected" (covers legacy `sk-` and modern `sk-proj-` formats; length varies)
- `xoxb-[A-Za-z0-9-]+` — "Slack bot token detected"
- `xoxp-[A-Za-z0-9-]+` — "Slack user token detected"
- `sk_live_[A-Za-z0-9]{20,}` — "Stripe live secret key detected"
- `SG\.[A-Za-z0-9_-]{22}\.[A-Za-z0-9_-]{43}` — "SendGrid API key detected"

**Do NOT block** `-----BEGIN CERTIFICATE-----` — public certificates are not secrets.

## Post-creation steps

After writing the three scripts:
1. `chmod +x` all three
2. Run `bash -n <script>` on each to verify syntax
3. If `shellcheck` is on PATH, run it on each and fix any warnings before proceeding

## Settings merge — the risky step

The `.claude/settings.json` merge must preserve existing configuration. Do this:

1. If `.claude/settings.json` exists, back it up: `cp .claude/settings.json .claude/settings.json.bak`
2. If it doesn't exist, create it as `{}`
3. Use `jq` to merge — not string manipulation, not sed, not a full rewrite. Example pattern (adapt as needed):

```bash
jq '.hooks.PreToolUse += [
  {
    "matcher": "Bash",
    "hooks": [
      {"type": "command", "command": "bash \"$CLAUDE_PROJECT_DIR/.claude/hooks/block-destructive-git.sh\"", "timeout": 5},
      {"type": "command", "command": "bash \"$CLAUDE_PROJECT_DIR/.claude/hooks/block-dangerous-bash.sh\"", "timeout": 5}
    ]
  },
  {
    "matcher": "Edit|Write|MultiEdit",
    "hooks": [
      {"type": "command", "command": "bash \"$CLAUDE_PROJECT_DIR/.claude/hooks/block-secrets.sh\"", "timeout": 5}
    ]
  }
]' .claude/settings.json > .claude/settings.json.tmp && mv .claude/settings.json.tmp .claude/settings.json
```

4. If the existing file already has a `hooks.PreToolUse` array with entries matching these same matchers pointing at these same scripts, do NOT append duplicates — detect and skip. If there are existing entries with the same matcher but different scripts, append as a new array entry (Claude Code runs all matching hooks).
5. After the merge, `jq . .claude/settings.json` to confirm the file is still valid JSON. If it is not, restore from `.bak` and stop.

## Verification

After everything is in place:
1. Print the final `.claude/settings.json` to the terminal for the user to eyeball
2. Print `ls -la .claude/hooks/` to confirm the three scripts exist and are executable
3. Remind the user: these hooks apply ONLY to this project. To use them in other projects, copy `.claude/hooks/` and merge the settings entries into each project's `.claude/settings.json`. Commit the `.claude/` directory to git so teammates inherit the same guardrails.
